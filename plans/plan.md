# git-review-tool 実装計画

## 目的

gitのコミット差分をHTMLで可視化し、hunk単位・行単位でセルフレビューできるツールを実装する。

## ゴール

- gitコミットの差分を取得してHTMLで表示できる
- diff2htmlを使って見やすい差分ビューを提供できる
- 各hunkに対してレビューコメントを保存できる
- 各行に対してコメントを保存できる
- 各hunkのレビュー完了状態を管理できる
- コミットハッシュが変わっても、同一hunkであればコメントと完了状態を引き継げる

## 想定スコープ

- 単一リポジトリを対象とするローカル実行ツール
- 対象リポジトリのディレクトリで必要時だけ起動することを前提とする
- 入力対象は特定コミット、またはコミット間差分
- UIはブラウザで閲覧できるHTMLベース
- 永続化先は対象リポジトリ配下のローカルファイルまたは軽量DBを前提とする

## 機能要件

### 1. 差分取得

- gitコマンドから対象コミットのdiffを取得する
- 対象指定方法を定義する
  - 単一コミット
  - 2コミット間
  - HEADと作業ツリーは将来拡張でもよい
- 取得したdiffをdiff2htmlに渡せる形式で保持する

### 2. 差分表示

- diff2htmlでHTMLを生成する
- ファイル単位、hunk単位で視認しやすく表示する
- hunkを識別できるIDを画面とデータの両方で扱う

### 3. コメント入力

- hunk単位でコメントを追加・更新できる
- 行単位でコメントを追加・更新できる
- コメント内容を再表示時に復元できる

### 4. レビュー状態管理

- hunkごとにチェック済みフラグを持つ
- UI上でチェック状態を変更できる
- 保存済み状態を再表示時に復元できる

### 5. hunk同一性判定

- 各hunkに安定したhash値を付与する
- コミットハッシュが変わっても、内容的に同じhunkを同一とみなせるようにする
- 同一判定に成功した場合、既存コメントとチェック済みフラグを引き継ぐ

## 非機能要件

- ローカルで簡単に起動できること
- 差分件数が多くても実用的な速度で表示できること
- データ保存形式が壊れにくく、バックアップしやすいこと
- 実装初期はシンプルさを優先し、過剰な抽象化を避けること

## 推奨アーキテクチャ

## 構成案

- 対象リポジトリ内で起動するCLI中心のアプリケーション
- 起動時に必要な処理だけを行い、常時起動は前提としない
- UI表示方式は以下のいずれかを採用する
  - 一時的なローカルHTTPサーバーを起動してブラウザで表示する
  - 静的HTMLを生成してブラウザで表示する
- バックエンド責務
  - git diff取得
  - hunk解析
  - hunk hash生成
  - コメントと状態の保存
- フロントエンド責務
  - diff2htmlによる差分表示
  - hunkコメント入力UI
  - 行コメント入力UI
  - 完了チェックUI

## データモデル案

### review_sessions

- id
  - レビューセッションを一意に識別するID。
- repository_path
  - レビュー対象リポジトリのローカルパスで、保存データの追跡や移行時の参照に使う。
- base_revision
  - 差分比較の基準となるコミットやリビジョン。
- target_revision
  - レビュー対象となるコミットやリビジョン。
- created_at
  - レビューセッションを作成した日時。
- updated_at
  - レビューセッションの内容を最後に更新した日時。

### hunks

- hunk_hash
  - hunkの同一性判定に使う安定した識別用ハッシュ値。
- file_path
  - このhunkが属する変更対象ファイルのパス。
- old_start
  - 変更前ファイル側でのhunk開始行番号。
- new_start
  - 変更後ファイル側でのhunk開始行番号。
- header
  - diff上のhunkヘッダー文字列。
- normalized_content
  - 同一判定用に正規化したhunk内容。

### hunk_reviews

- review_session_id
  - どのレビューセッションに属するhunkレビューかを示すID。
- hunk_hash
  - 対象hunkを特定するためのハッシュ値。
- comment
  - hunk全体に対するレビューコメント本文。
- checked
  - そのhunkのレビュー完了状態を表す真偽値。
- updated_at
  - hunkレビューを最後に更新した日時。

### line_comments

- review_session_id
  - どのレビューセッションに属する行コメントかを示すID。
- hunk_hash
  - 行コメントが属するhunkを特定するためのハッシュ値。
- line_key
  - hunk内のどの行に紐づくコメントかを識別するキー。
- comment
  - 対象行に対するレビューコメント本文。
- updated_at
  - 行コメントを最後に更新した日時。

## hunk hash設計案

- hash生成対象
  - file_path
  - hunk headerの構造情報
  - 追加行と削除行の正規化済み内容
- hash生成時の注意
  - 行番号そのものには過度に依存しない
  - 空白差分や改行差異の扱いを事前に決める
  - rename時の扱いは初期版で制約を明示する

## 実装ステップ

### Phase 1: 土台作成

- 使用技術を決定する
- プロジェクト構成を作成する
- diff取得の最小CLIを実装する
- diff2htmlで静的HTMLを出力するところまで作る

### Phase 2: hunk解析と識別

- diffからファイル、hunk、行情報を抽出する
- hunk hash生成ロジックを実装する
- 画面上の各hunkに一意な識別子を紐づける

### Phase 3: 永続化

- コメントとチェック状態の保存先を実装する
- hunkコメント保存を実装する
- 行コメント保存を実装する
- 再表示時の復元を実装する

### Phase 4: レビューUI

- hunk単位コメント入力UIを実装する
- 行単位コメント入力UIを実装する
- チェック済みフラグ操作UIを実装する
- 保存結果が即時反映されるようにする

### Phase 5: 引き継ぎロジック

- 新しいdiff読み込み時に既存hunk hashと照合する
- 一致したhunkへコメントと状態を引き継ぐ
- 引き継ぎ失敗時の扱いを定義する

### Phase 6: 仕上げ

- 操作フローを見直す
- READMEと使い方を整備する
- エラーハンドリングを追加する
- テストを追加する

## MVP定義

初期リリースでは以下を満たせばよい。

- 単一コミットのdiffをHTML表示できる
- hunk単位コメントを保存できる
- hunkのチェック済みフラグを保存できる
- 同一hunk判定により再レビュー時に状態を復元できる

行単位コメントはMVPに含めてもよいが、工数次第でPhase 2以降へ分離可能。

## 技術選定のたたき台

- バックエンド
  - Node.js + TypeScript
- フロントエンド
  - シンプルなサーバーサイドHTML、または軽量SPA
- 差分表示
  - diff2html
- 永続化
  - 初期版は対象リポジトリ配下のJSONファイル、拡張時はSQLite

JSONで始める理由は、実装速度と可搬性を優先できるため。保存先は対象リポジトリ配下の専用ディレクトリを想定し、複数レビューの検索や同時編集が必要になった時点でSQLiteへ移行する。

## 保存先方針

- レビュー用データは対象リポジトリ配下の専用ディレクトリに保存する
- 候補ディレクトリ名は .git-review-tool/ とする
- 差分HTML、レビュー状態、コメントをこの配下で管理する
- git管理対象に含めるかどうかは運用方針として別途決める

## テスト計画

- 単体テスト
  - diff解析
  - hunk hash生成
  - 引き継ぎ判定
  - 永続化読み書き
- 結合テスト
  - git diff取得から画面表示まで
  - コメント保存から再表示復元まで
- 手動確認
  - 複数ファイル差分
  - renameを含む差分
  - 大きいdiff
  - コメント引き継ぎの成否

## リスクと対策

- hunk hashの安定性が不足すると引き継ぎ精度が落ちる
  - 対策: 実データで複数パターンを検証し、正規化ルールを固定する
- 行単位コメントはdiff再生成時の行対応付けが難しい
  - 対策: 初期版では制約を明示し、hunk内相対位置ベースで管理する
- diff量が多いとHTML描画が重くなる
  - 対策: ファイル折りたたみ、遅延描画、表示件数制御を検討する
